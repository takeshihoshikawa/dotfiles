# Morning

1日の Introduction を作る。`daily-report recap` が同じ項目へ Results / Discussion を返せるよう、計画を当日の日報へ永続化する。

## 対象日

- 引数に `tomorrow` があれば明日、それ以外は今日。
- タイムゾーンは `Asia/Tokyo`。

## 朝夕の対応

| morning | daily-report recap |
|---|---|
| 文脈 | 目標への寄与 |
| プロジェクト優先順位 | 実際のプロジェクト配分と逸脱 |
| 今日の主題・問い | 得られた答え・残った問い |
| 期待する成果 | 実際の成果 |
| 本人／委任の分担 | 本人の活動／委任成果 |
| エフォート予算 | エフォート実績 |
| 制約・リスク | 計画差と理由 |
| 完了条件 | 達成証拠 |

## 週次との対応

`weekly-review`が作成した対象週の計画を、その日の既定方針として具体化する。

- 対象日を含む週の月曜日を求め、`weekly/weekly-{Monday}.md`の`## 週の計画`を読む。
- 現行形式がない場合だけ、直前週の`## 来週の計画`または`## 来週の重点`を移行用の参考情報として読む。
- 週次で決めたプロジェクトの役割、期待成果、本人と委任の分担、エフォート予算を、当日の利用可能時間へ落とし込む。
- 新しい締切、体調、依存関係、前日の結果などの根拠がなければ、週次の役割を朝だけで変更しない。
- 週次計画から外れる提案をする場合は、変更内容と理由を明示し、`daily-report recap`で追跡できる形にする。

## 収集

可能なものは並列実行する。失敗した情報源だけを省略し、全体を止めない。

### リポジトリ同期

`~/work/projects/`直下の全gitリポジトリと`~/dotfiles`をfetchする。cleanかつbehindのみならpull --rebaseする。改行区切りは`while IFS= read -r repo`で処理する。

- pullしたリポジトリだけ簡潔に報告する。
- fetch失敗、dirty、divergedは`/sync-repos`で別途扱う。

### プロジェクトレーダー

Obsidianを起動し、`~/work/projects/admin/scripts/project_radar.py`を実行する。失敗しても続行し、通常は出力しない。

更新後、`projects/*.md`の`status: active`と`status: waiting`をすべて読む。まずfrontmatterの次の項目を一覧化し、判断に必要なプロジェクトだけ本文を読む。

- `status`
- `current_phase`
- `next_action`
- `concern`
- `last_touched`
- `open_tasks`
- `overdue_tasks`
- `waiting_tasks`
- `last_completion`

対応するタスク、GitHub Issues、直近の日報・コミットも、優先順位の根拠として照合する。

### 文脈

- `notes/goals.md`
- 対象週の`weekly/weekly-{Monday}.md`にある`## 週の計画`
- 対象日より前の最新の`daily/YYYY-MM-DD.md`
- 前日の日報では特に`## 気づき`と`## 次への引継ぎ`を読む。

### 予定

Google Calendarから対象日と翌日の予定を別々に取得する。必ず明示的な日付境界と`Asia/Tokyo`を指定し、開始時刻順に整理する。

### タスクとGitHub Issues

- `obsidian tasks todo format=json`でvault全体の未完タスクを取得する。
- CLI不調時はvaultルートで`rg`を使い、`templates/**`と`courses/**`を除外する。
- `#waiting`は表示しない。
- 期限切れ、本日期日、日付なしNextを分ける。
- GitHubのopen Issueはプロジェクト別に要約する。取得失敗時は省略する。

## 計画を作る

収集結果から次を提案する。単なるタスク列挙ではなく「今日の終わりに何が変わっていれば成功か」を定義する。

### 文脈

- 長期目標との接続を1–2点に絞る。
- 当週の週次計画から引き継ぐ役割・期待成果・エフォート予算を要約する。
- 前日の結果・気づき・引継ぎを3–5項目で要約する。

### プロジェクト優先順位

個別タスクを選ぶ前に、全プロジェクトへその日の役割を提案する。
まず週次計画の役割を既定値とし、当日の新しい証拠に応じて具体化または変更する。週次で待機としたプロジェクトを、根拠なく日次で再活性化しない。

| 役割 | 意味 | 原則 |
|---|---|---|
| 集中 | 深い作業を割り当て、主要成果を狙う | 最大2件。主軸は原則1件 |
| 期限対応 | 締切、外部約束、短時間の解除作業を処理する | 必要最小限 |
| 維持 | 小さな前進、確認、次の一手の明確化だけ行う | 時間上限を置く |
| 待機 | 外部待ち、前提未成立、今は動かさない | 原則作業しない |

次の観点を総合し、単純な点数合計では決めない。

1. 長期目標の4本柱への寄与と、今期の戦略的位置づけ
2. 締切、外部コミットメント、他者を止めている依存関係
3. 今の一手で重要な不確実性を解消できるか
4. 投稿直前、検証直前など、完了に近い成果の回収可能性
5. 最近の進展と切替コスト。勢いだけで惰性的に続けない
6. `concern`、overdue、waiting、放置期間。ただしタスク数や古さだけで優先しない
7. 当日の利用可能時間、体調、会議、研究・事務・教育の必要配分

提案には必ず次を含める。

- 集中プロジェクトと、今日それを選ぶ理由
- 期限対応・維持するプロジェクトと時間上限
- 待機するプロジェクトと「今日はやらない」理由
- 優先順位を変える条件
- 全体を見た明示的な「やらないこと」

ユーザーが明示的に決めた優先順位を、客観指標だけで上書きしない。矛盾があれば根拠を示して提案し、決定は確認する。

### 今日の主題

1日の中心となる主題を原則1つにする。必要なら副次的な主題を1つまで置く。

### 今日の問い

その日の作業で答えたい問い、確定したい判断、解消したい不確実性を書く。作業手順を問いにしない。

### 期待する成果

最大3–5件に絞り、各成果に実施主体と完了条件を付ける。
週次の期待成果から、その日に回収できる最小の到達点を選ぶ。週次にない成果を追加する場合は、追加理由を明示する。

- `[本人]`：思考、意思決定、対話、執筆、会議など本人が担う。
- `[委任]`：エージェントへ任せる調査、実装、検証、整理。
- 共同作業は本人が確定する判断と、委任する処理を分ける。

### エフォート予算

本人の利用可能時間だけを、次の3区分へ配分する。

| 区分 | 内容 |
|---|---|
| 研究 | 研究、研究のための学習、論文、研究プロジェクト |
| 事務 | 大学事務、調達、業務管理、朝の計画 |
| 教育 | 授業、学生対応、採点、教材作成 |

- カレンダー、勤務時間、昼休みを考慮して時間と割合を提案する。
- 週次のエフォート予算と、それまでの実績が分かる場合は残り配分を判断材料にする。
- エージェントの処理時間は予算へ含めない。
- 合計は本人の利用可能時間と一致させる。

### 制約・予定

会議、締切、昼休み、体調、移動、勤務終了時刻、依存関係を記載する。必要なら停止条件も書く。

## 期限切れタスクのトリアージ

- 今日モード：前日以前の期限切れだけを対象にする。
- tomorrowモード：本日期日のタスクも対象にする。
- 各タスクについて`完了(c) / 延ばす(YYYY-MM-DD) / 削除(d)`をまとめて確認する。
- 完了は`obsidian task ref="path:line" done`、延期はdue日を編集、削除は該当行を削除する。
- 優先順位は明示依頼がある場合だけ変更する。

## ユーザー確認と保存

1. プロジェクト優先順位案を提示し、集中・期限対応・維持・待機の判断を確認する。
2. 計画案とタスク候補を提示し、「今日の主題・期待成果・エフォート予算」を確認する。
3. 選んだタスクへ対象日のdueを付けるか確認する。
4. 承認後、`daily/YYYY-MM-DD.md`へ`## 朝の計画`を作成または置換する。他のセクションは保持する。
5. ファイルがなければfrontmatterと`## 朝の計画`を作る。
6. 同日に再実行した場合は重複追記せず、既存の朝の計画を更新する。
7. Markdown本文でリストをネストする場合は、Obsidianの設定に合わせて子項目を半角4スペースでインデントする。YAML frontmatterは通常のYAMLインデントを使う。

## チャット出力

1. 長期目標のうち当日に関係する部分
2. 当週の週次計画と、必要なら変更理由
3. 前日の結果・気づき・引継ぎ
4. 全プロジェクトの優先順位案
5. 対象日と翌日の予定
6. 期限切れ、本日期日、Next、GitHub Issues
7. 朝の計画案
8. トリアージまたは計画確認の質問

簡潔に表示し、内部のファイル参照と行番号はタスク操作に必要な間だけ保持する。
