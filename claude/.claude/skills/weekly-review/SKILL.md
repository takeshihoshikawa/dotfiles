# Weekly Review

週の`Introduction → Results / Discussion`を閉じ、次週のIntroductionを作る。

```text
週の計画 → morning → daily-report recap → 週の振り返り → 次週の計画
```

## 対象週

- 明示された日付、`this week`、`last week`、`0`、`-1`を優先する。
- 指定がなければ月〜水は前週、木〜日は今週を対象週とする。
- 週は`Asia/Tokyo`の月曜00:00〜日曜23:59:59とする。
- 対象週の次の月曜〜日曜を次週とする。

## 週報ファイルの意味

`weekly/weekly-{週の月曜日}.md`を、その週の計画と振り返りの正本とする。

- 週の開始前：`## 週の計画`を作る。
- 週の終了時：同じファイルへ`## 週の振り返り`を作る。
- 同日に再実行してもセクションを重複させず、対象セクションを置換する。
- 過去の週報は移行しない。
- 対象週ファイルに`## 週の計画`がなく、前週ファイルに旧形式の`## 来週の計画`がある場合は、計画比較の補助証拠として読む。予定値を補作しない。

## 共通原則

### 本人は時間ベース

- 日報の本人活動区間だけを研究・事務・教育へ集計する。
- 並行区間は和集合で集計し、二重計上しない。
- 時間未計測の断続対応、昼休み、私用、エージェント処理を含めない。
- gitコミット時刻から本人の作業時間を推定しない。

### エージェントは成果ベース

- 日報の`### 委任した処理・成果`をプロジェクト別・成果単位で集約する。
- 実行時間、夜間の経過時間、並列数を本人エフォートへ含めない。
- 未検証、本人未採用、処理中の成果は状態を明記する。

### プロジェクトは配分で評価する

- 週の主軸へ本人の深い作業が配分されたかを見る。
- 期限対応・維持が設定範囲に収まったかを見る。
- 待機対象への意図しない流入とコンテクストスイッチを確認する。
- 新しい証拠に基づく優先順位変更と、単なる逸脱を区別する。
- エージェント成果はプロジェクト進展として扱うが、本人の主軸数や時間へ加算しない。

### Markdown形式

- Markdown本文の子リストは半角4スペースでインデントする。
- YAML frontmatterは通常のYAMLインデントを使う。

## Phase 1: 収集

可能なものは並列実行する。失敗した情報源だけを省略し、全体を止めない。

### 対象週の証拠

- 対象週の`daily/YYYY-MM-DD.md`を月〜日で読む。特に次を抽出する。
    - `## 朝の計画`
    - `### 本人の活動`
    - `### 委任した処理・成果`
    - `## 計画との比較`
    - `## プロジェクト配分`
    - `## エフォート配分`
    - `## 気づき`
    - `## 次への引継ぎ`
- `weekly/weekly-{対象週月曜}.md`の`## 週の計画`を読む。
- 旧形式しかない場合は、前週ファイルの`## 来週の計画`と`## 来週の重点`を読む。
- 対象週内に完了したObsidianタスクを補助証拠として読む。
- 対象週の成果確認に必要な場合だけgitログとclosed GitHub Issueを照合する。

現在のプロジェクト状態や対象週後のコミットを、対象週の成果として混入させない。

### 次週計画の材料

- `notes/goals.md`
- `tasks.md`の未完、期限切れ、次週期限、`#waiting`、inbox
- 今後2週間のGoogle Calendar
- 今後2週間の担当授業
- open GitHub Issues
- `projects/*.md`の`status: active`と`status: waiting`

`~/work/projects/admin/scripts/project_radar.py`を実行し、週次レビュー単独でも客観指標を最新化する。morning実行済みと仮定しない。

リポジトリはfetchし、cleanかつbehindのみならpull --rebaseする。dirty、ahead、divergedは報告し、別目的の変更を自動コミットしない。

## Phase 2: 週の振り返り

### 2.1 計画との比較

対象週の`## 週の計画`にある期待成果を1件ずつ実績と照合する。

| 判定 | 意味 |
|---|---|
| 達成 | 完了条件を満たした |
| 一部達成 | 重要部分は進んだが未完 |
| 未着手 | 着手証拠がない |
| 変更 | 新しい証拠や制約に基づき別成果へ切り替えた |
| 想定以上 | 完了条件を超えて進んだ |

週の計画がなければ「週の計画なし」と記載し、予定値を作らない。

### 2.2 プロジェクト配分

週の優先順位と、各日の`## プロジェクト配分`を照合する。

- 本人の週間主軸が継続したか。
- 日ごとの集中対象が分散しすぎなかったか。
- 期限対応・維持・待機が意図どおりだったか。
- 委任レーンが本人のコンテクストスイッチを増やさず成果を出したか。
- 次週に優先順位を変える証拠が得られたか。

### 2.3 エフォート配分

- 日報から本人の研究・事務・教育の実績時間を合計する。
- 週の予算があれば予定／実績／差を示す。
- 日報に時間不明があれば、週次レビューで推測せず「時間未計測」とする。
- 健康、休暇、移動など実績解釈に必要な制約を時間集計とは分けて残す。

### 2.4 成果とDiscussion

- 本人の主要成果と委任成果を分け、プロジェクト別に意味のある成果へ束ねる。
- 研究・事務・教育の責務別に、達成・未達と理由を簡潔にまとめる。
- `## 気づき`を横断し、結果の意味、計画差、集中の質、健康・環境上の制約、次週への含意をDiscussionとして書く。
- ユーザーへ次を確認する。
    - 振り返りの事実関係と解釈
    - 今週のひと言
    - 中長期目標との接続

確認後に振り返りを確定する。まだ保存しない。

## Phase 3: 次週のポートフォリオ

タスクを曜日へ置く前に、全active／waitingプロジェクトの次週の役割を提案する。

| 役割 | 原則 |
|---|---|
| 本人の主軸 | 原則1件。本人の深い作業を継続して割り当てる |
| 期限対応 | 外部約束、締切、短時間の解除作業に限定する |
| 維持 | 小さな確認・前進だけ。週の時間上限を付ける |
| 待機 | 外部待ち、前提未成立、今は動かさない |

委任レーンは本人の主軸と分け、原則2件以内で提案する。各レーンに成果とレビュー条件を付け、エージェント時間を週のエフォート予算へ含めない。

必ず次を提案する。

- 次週の主題
- 次週に答える問い
- プロジェクト優先順位と変更条件
- 明示的な「今週はやらないこと」
- 期待成果3〜5件。`[本人]`／`[委任]`と完了条件を付ける
- 本人の研究・事務・教育エフォート予算
- カレンダー、授業、締切、健康、依存関係、停止条件

morningはこの週次優先順位を既定値として日次へ具体化する。週次判断を変える必要がある場合は、新しい証拠と理由を次の日報へ残す。

## Phase 4: タスクと曜日配置

### タスク提案

次から、次週に実行可能なタスクだけを提案する。

1. 未達・持越し・`## 次への引継ぎ`
2. 次週の期待成果を実行可能にする不足タスク
3. 次週の授業・予定の準備とフォローアップ
4. inboxの分類

既に同じ実行単位のタスクがあれば追加しない。期待成果を細かいチェックボックスへ分解しすぎない。

ユーザーの承認後、AGENTS.mdの正本規則に従って保存する。

- 計画起点のプロジェクトタスク：vaultルート`tasks.md`の`## projects`
- 大学・機関起点：vaultルート`tasks.md`の`## admin`
- 授業・学生起点：vaultルート`tasks.md`の`## teaching`
- 分類不明：vaultルート`tasks.md`の最終セクション`## inbox`
- 会議起点：該当meeting note。`tasks.md`へ複製しない
- project note：タスクチェックボックスを追加しない

`#project/{kebab-case}`は`projects/{kebab-case}.md`と一致させる。タスク追加、完了、延期、削除、`#waiting`解除は必ず確認後に実行する。

### 研究コアブロック

- 次週のGoogle Calendarに`研究コアブロック`が1〜2件あるか確認する。
- 未登録なら、休日・会議・授業を避け、平日08:00〜10:30を優先して提案する。
- 承認後だけGoogle Calendarへ作成する。

### 曜日配置

プロジェクト優先順位、期待成果、エフォート予算を確定してから、予定とタスクを曜日へ配置する。

- 主軸の連続ブロックを先に置く。
- 期限対応を必要最小限にまとめる。
- 維持プロジェクトは設定した時間上限を超えない。
- 予定が崩れても期待成果と優先順位が残る粒度にする。

次週計画全体を提示し、ユーザー確認を得る。

## 対話ゲート

確認は3回にまとめる。

1. 週の振り返りと解釈
2. 次週のポートフォリオ、期待成果、タスク、カレンダー
3. 対象週の振り返りと次週計画の保存

ユーザーが終了を選んだ場合は、その時点で安全に終了する。
