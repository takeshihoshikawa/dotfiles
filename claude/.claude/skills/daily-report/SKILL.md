# Daily Report

`morning`の Introduction に対する Results / Discussion を作る。本人の時間とエージェントの成果を混同しない。
標準化されたrecapの各節は、`weekly-review`が計画対実績、プロジェクト配分、本人エフォート、委任成果を週単位で集約するための入力にもなる。

## 対象日とモード

- 日付指定があればその日、なければ今日。タイムゾーンは`Asia/Tokyo`。
- 引数なし：appendモード。
- `recap`：1日を再構成するrecapモード。

## 共通原則

### 本人は時間ベース

- 本人が実際に活動した、根拠のある区間だけを時刻付きで記録する。
- 明示時刻、カレンダー上の参加予定、会話や操作記録から妥当に分かる時刻を使う。
- gitコミット時刻やエージェントの実行時間から本人の労働時間を推定しない。
- 並行区間は和集合で集計し、二重計上しない。
- 断続的な指示・確認は実時間が分かる場合だけ加算する。不明なら「時間未計測」として集計外にする。
- 不明な30分以上の区間は確認する。分からない時間を隣の作業へ吸収して稼働時間を増やさない。

### エージェントは成果ベース

- 調査、実装、検証、文書化などをプロジェクト別・成果単位でまとめる。
- 実行開始・終了時刻、夜間の経過時間、並列数は日報本文へ原則記載しない。
- コミットとGitHub Issueは成果の証拠、順序、到達点の把握に使う。
- 本人が採用していない試行や未検証結果は「試行」「未検証」と明記する。
- エージェントの処理時間を本人の稼働時間やエフォートへ含めない。

### Markdown形式

- Markdown本文でリストをネストする場合は、Obsidianの設定に合わせて子項目を半角4スペースでインデントする。
- YAML frontmatterは通常のYAMLインデントを使う。

### エフォート区分

本人の活動だけを分類する。

| 区分 | 内容 |
|---|---|
| 研究 | 研究、研究のための学習、論文、研究プロジェクト |
| 事務 | 大学事務、調達、業務管理、朝の計画 |
| 教育 | 授業、学生対応、採点、教材作成 |

## appendモード

1. 会話から新しい本人活動、委任成果、気づきを抽出する。
2. 対象日の`daily/YYYY-MM-DD.md`に`## 日中メモ`がなければ作る。
3. 重複を避けて末尾へ追記し、確認なしで終了する。

形式：

```markdown
## 日中メモ

- HH:MM–HH:MM [本人/研究] **作業内容**
    - 詳細
- [委任/研究/project-name] **確認できた成果**
    - 詳細
- [気づき] 内容
```

終了時刻や区分が不明なら捏造せず、省略または暫定表記にする。

## recapモード

次の順で進める。

1. 収集
2. 再構成してユーザー確認
3. 保存
4. 完了タスクの確認と処理
5. リポジトリの片づけ

### 1. 収集

可能なものは並列実行する。

- 対象日の`daily/YYYY-MM-DD.md`全文。特に`## 朝の計画`と`## 日中メモ`。
- Google Calendarの対象日の時刻付き予定。終日イベントは除外する。
- 対象日にcloseされた本人作成のGitHub Issues。
- `~/work/projects/`直下の全gitリポジトリと`~/dotfiles`の対象日コミット。
- 会話履歴にある本人の明示時刻、体調、判断、指示、成果。

gitは各リポジトリでfetchし、cleanかつbehindのみならpull --rebaseしてからlogを読む。改行区切りは`while IFS= read -r repo`で処理する。取得失敗は報告して続行する。

### 2. 再構成

#### 本人の活動

- 根拠のある活動を時刻順に並べる。
- 会議・授業・受講はカレンダーで補完する。
- 昼休み、早退、体調不良、移動など実績解釈に必要な制約を残す。
- 30分以上の不明区間、退勤後の本人関与、分類が曖昧な活動だけをまとめて確認する。

#### 委任した処理・成果

- コミットを時刻順に転記しない。
- 複数コミットを「研究設計の具体化」「パイプライン改善」「検証完了」など意味のある成果へ束ねる。
- プロジェクト、成果、到達状態、重要な訂正や未解決点を書く。
- GitHub closed Issueも同じ成果へ統合し、重複させない。

#### 計画との比較

`## 朝の計画`の期待成果を1件ずつ、実績と照合する。

| 判定 | 意味 |
|---|---|
| 達成 | 完了条件を満たした |
| 一部達成 | 重要部分は進んだが未完 |
| 未着手 | 着手証拠がない |
| 変更 | 意図的に別の成果へ切り替えた |
| 想定以上 | 完了条件を超えて進んだ |

計画がない日は予定値を捏造せず、「朝の計画なし」と記載する。

#### プロジェクト配分

朝の`### プロジェクト優先順位`と、本人の活動・委任成果を照合する。

- 集中対象へ本人の深い作業が配分されたか
- 期限対応・維持が設定した範囲で収まったか
- 待機対象へ意図しない作業が流入しなかったか
- 新しい証拠に基づく意図的な優先順位変更だったか、単なる逸脱だったか
- 複数プロジェクトへの分散で主要成果が弱まっていないか

エージェント成果はプロジェクト進展として記録するが、本人のエフォートへは加算しない。計画と異なるプロジェクトで大きな成果が出た場合も、理由と次の優先順位への含意を書く。

#### エフォート配分

1. 本人の活動区間を研究・事務・教育へ分類する。
2. 重複区間を除き、区分別時間と合計を計算する。
3. 朝の予算があれば予定／実績／差を示す。
4. エージェント成果、時間未計測の断続対応、昼休みは含めない。

#### Discussion

- `## 気づき`に、結果の意味、計画差の理由、今後への含意を2–3段落で書く。
- `## 次への引継ぎ`に、翌朝の文脈となる判断、未解決の問い、持越しを最大3–5件書く。
- 新しいタスクチェックボックスは日報へ作らない。

下書きをユーザーへ提示し、確認を得てから保存する。

### 3. 保存

対象日のファイルを全文更新する。

- `## 朝の計画`はそのまま保持する。
- `## 日中メモ`は最終構成へ統合し、重複するなら削除する。
- 同日に再実行しても各セクションを重複させない。

完成形式：

```markdown
---
date: YYYY-MM-DD
tags:
  - daily
---

## 朝の計画

（morningで承認した内容。なければ省略）

## やったこと

### 本人の活動

- HH:MM–HH:MM **作業内容**
    - 詳細

### 委任した処理・成果

#### project-name

- **成果**
    - 到達点・証拠・未解決点

## 計画との比較

| 期待した成果 | 結果 | 判定 |
|---|---|---|

## プロジェクト配分

| プロジェクト | 朝の役割 | 実際の関与・成果 | 評価 |
|---|---|---|---|

- 集中を守れたか：
- 優先順位を変える新しい証拠：
- 意図しない分散：

## エフォート配分

| 区分 | 予定 | 実績 | 差 |
|---|---:|---:|---:|
| 研究 |  |  |  |
| 事務 |  |  |  |
| 教育 |  |  |  |

- 本人稼働時間：
- 時間未計測の断続対応：
- エージェント処理は集計対象外

## 気づき

（結果の解釈、計画差の理由、今後への含意）

## 次への引継ぎ

-
```

### 4. 完了タスク

「実際の成果」から完了と判断できる未完タスクをvault全体で検索して候補を提示する。確認後に`obsidian task ref="path:line" done`を使う。新規タスクは追加しない。

### 5. リポジトリの片づけ

全対象リポジトリを確認する。

| 状態 | 処理 |
|---|---|
| dirty | diffを確認し、一貫した安全な変更だけcommit+push。判断不能なら報告 |
| ahead | push |
| pullable | pull --rebase |
| diverged | 触らず報告 |

WIP、デバッグ痕跡、秘密情報、複数目的が混在する変更は自動コミットしない。結果はチャットへ報告し、日報本文には書かない。
