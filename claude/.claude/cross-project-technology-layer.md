# プロジェクトをまたぐ技術資産の再利用

「同じロジックを別プロジェクトでまた書きそうになった」時に参照する判断フィルタ。単一プロジェクトの構造はObsidian vaultの`notes/research-project-setup.md`が正本、本ノートは複数プロジェクト間の再利用を扱う。

実例: pine-wilt-shizuoka プロジェクトでの `lidar-toolkit`/`lidar-toolkit-r` 切り出し（2026-07-09）。

## いつこれを思い出すか

- 新しいプロジェクトを書いていて「これ、前にも書いた気がする」と思ったら
- 既存プロジェクトを読んでいて、別プロジェクトの実装と同じロジックが3つ目のリポジトリでも独立に書かれているのに気づいたら

**これはチェックリストではない。** 毎回の新規プロジェクト立ち上げ時に強制適用するものではなく、上記の兆候が実際に出た時にだけ思い出す。

## 背景（なぜこの問題が起きるか）

関連プロジェクトは、顧客・資金源の違いによって断続的に別リポジトリとして進む（例: 委託研究の納品物リポジトリ、その後の自主研究、科研費の予備実験は、技術的に地続きでも別案件）。技術的には連続した知見の蓄積なのに、放っておくと新しいプロジェクトのたびに過去リポジトリから使える処理を手作業で「発掘」して移植することになる。これは非効率なだけでなく、移植の過程で改善が過去プロジェクトへ還元されない・同じ罠に複数回落ちる、という損失を生む。

## 抽出の判断基準

- 新しい抽象化は「実際に2回以上の重複が実証された時」のみ行う（`data-analysis-coding-conventions.md`の「2回以上使う処理は関数化する」と同じ閾値を、プロジェクト間の再利用にも適用する）
- 1プロジェクトでしか使っていない処理を、将来使うかもしれないからと先回りして共有化しない。推測でインターフェースを設計すると、実際の2つ目の利用が来た時にほぼ作り直しになる
- 抽出対象になるのは「技術層」（センサー・ツール固有で目的非依存な処理。例: 衛星データ取得、点群処理、UAV画像処理）のみ。「案件固有」（特定の顧客・AOI・データスキーマに依存する処理）は抽出せず、消費側プロジェクトに残す
- 目的別の手法（例: 特定の検出アルゴリズム）がサイト・案件をまたいで安定して使われている場合、それは既にその手法の「本体プロジェクト」が担っていることが多く、わざわざ別リポジトリに切り出す必要はない（技術層と案件層の中間に「目的層」を無理に作らない）

## wrapper化する価値の基準

関数化・パッケージ化する価値があるのは「APIの言い換え」ではなく「痛い目にあって得た決定を構造的に埋め込むもの」。判定基準: **「このwrapperを外して生のAPI/ロジックを直接呼んだら、何が静かに壊れるか？」に具体的な答えがあるか。** 答えがある（実測に基づく既定値、既知の罠の回避、座標系の取り違え防止等）なら包む価値がある。答えが無い（ただの改名）なら、それは「寄せ集め」であり抽出しない。

## 実装の型（個人開発向けの軽量版）

- 外部消費者がいないので semver・CI は組まない。gitコミットハッシュを固定して参照する
  - Python: `uv add "pkg @ git+https://github.com/.../repo.git@<commit>"`
  - R: `remotes::install_github("owner/repo", ref = "<commit>")`
  - 更新は消費側が明示的にpinを上げることで行う。自動追従はしない
- 複数言語が絡む場合は**言語ごとに別リポジトリ**に分ける。対になる名前を付ける（例: `xxx-toolkit` / `xxx-toolkit-r`）
  - 理由: コーディングエージェントは1セッションで通常1つの作業ディレクトリを見る。モノレポで複数のパッケージング体系（`pyproject.toml`と`DESCRIPTION`等）が同居すると、「テストを実行して」のような指示が曖昧になり、コンテキストにも無関係な言語のノイズが混ざる
- 各リポジトリに `docs/provenance.md` を置き、何を・どこから・なぜ採用したか（**採用しなかったものとその理由**も）を記録する。移行時点のスナップショットで終わらせず、以後の追加もここに書き足し続ける生きた台帳にする。「あの機能どこだっけ」を将来にわたって根絶するのが目的
- 各リポジトリの `CLAUDE.md` に「何を足すか・足さないかの基準」（上記「抽出の判断基準」の要点）と「消費側からの参照方法」を明文化し、将来別セッションのエージェントが単独で判断できるようにする
- `renv`/ロックファイルはライブラリ側リポジトリには含めない。`DESCRIPTION`のImports・`pyproject.toml`のdependenciesが依存宣言であり、厳密なバージョン固定は利用側プロジェクトの責務とする

## 統合元リポジトリの扱い

**削除・リネームしない。** README冒頭に凍結・統合先へのリンクを追記してそのまま残す（過去の引用・リンクを壊さないため。他プロジェクトのCLAUDE.mdや報告書が旧リポジトリ名を引用している場合がある）。データ実体が統合元側にある場合（例: S3上の配布データ）は無理に移設せず、そのまま参照を続けてよい。

## 見つかった副産物の扱い

移植作業（特に実データでのsmoke test）は、元実装が真に検証されていなかった潜在バグを見つける機会になることがある（例: 少数サンプルでのみ壊れる集計ロジック）。見つかったら、統合先リポジトリの `docs/provenance.md` に修正内容と発見の経緯を記録する。元の凍結リポジトリは無修正のまま残してよい（凍結の目的は歴史的スナップショットの保存であり、遡及的な品質保証ではない）。
